## Escalations for PA (flagged, not enshrined)
1. **`.sorted()` unimplemented at runtime** (§59.8): `.entries().sorted()` / `.keys().sorted()` — AND a plain array `[3,1,2].sorted()` — throw `TypeError: ....sorted is not a function`. §59.8 names `.sorted()`/`.sortedBy(fn)` as the deterministic-iteration stabilizer, but impl#1 doesn't wire it on scrml arrays (map-view or plain). General array-method gap; cases avoid `.sorted()` and assert row COUNT not order.
2. **Map cell snapshot leaks impl#1 HAMT internals**: the OQ3 conformance snapshot returns the raw map structure (`__scrml_map`/`_root`/`bitmap`/`nodes`) for a map cell, not a value-canonical form. Adapter-side follow-on (value-canonicalize map/set cells in `snapshot()`) would enable impl-neutral direct `state` assertions on map cells; until then, derived-scalar projections are the impl-neutral path (used here).
3. **W-MAP-* severity: SPEC says Info, impl emits `warning`** (§59.11/§59.8): W-MAP-DUPLICATE-LITERAL-KEY and W-MAP-STRUCT-KEY-LITERAL emit §34 severity `warning`, but §59.11's table + §59.8 explicitly state Info-level. Both are non-fatal (warnings stream), so it's a severity-LABEL divergence, not a partition one. Cases assert code PRESENCE only (no severity) to avoid enshrining warning-over-info.
4. **W-MAP-ITERATION-ORDER does not surface via the harness**: the documented `<each in=@m.entries()>` on a non-@ordered map without `.sorted()` (per lint-w-map-iteration-order.js) did NOT emit the code through the conformance adapter's compile path. Possible adapter lint-coverage gap or a stricter runtime trigger. NOT authored (can't go green); flagged. The other two W-MAP-* lints cover the family.
5. **E-MAP-BRACKET-WRITE inline-handler gap**: `@m[k]=v` fires E-MAP-BRACKET-WRITE in a function body / top-level logic, but a bracket-write inside an inline event handler (`onclick=${@m[k]=v}`) is NOT caught. Diagnostic-coverage gap; the case uses the function-body form.
6. **Refinement codes are E-CONTRACT-*, not E-REFINEMENT-*** (§53.11): the list item 4 illustratively named `string(pattern(/…/))` + an `E-REFINEMENT-*` family. The actual §53 syntax is `base-type(predicate)` with named shapes (`string(email)`) and the code family is `E-CONTRACT-001` (compile literal) / `-001-RT` (runtime boundary) / `-002` (unknown shape) / `-003` (external ref). (The lone `E-REFINEMENT-NO-DEFAULT` in the impl is unrelated.) Cases authored against the real §53. Not a divergence — just a list-description vs impl naming note.
7. **Refinement runtime boundary check: FUNCTION-boundary enforced, direct reactive-cell reassignment NOT** (§53.3.3/§53.4): a violating value passed to a refined function PARAMETER throws `E-CONTRACT-001-RT` and retains the prior value (verified, case authored). BUT a direct reactive-cell reassignment `@amount = @amount * 3` (→15000, violating `<10000`) is APPLIED with no runtime check. §53.3.4 says "a number MAY NOT be assigned to a `number(>0 && <10000)` variable without a boundary check," yet the reactive-cell path elides it. Zone-classification question (is a reactive-cell reassignment boundary-zone or trusted-zone?) — flagged for a PA/user ruling, not enshrined (the authored RT reject-case uses the function boundary, which IS enforced).
