# ss52 — non-reactive local map/set method lowering (the (c) bug) — VERIFY-FIRST · Road-B-critical

**Fill-note:** value-native maps/sets **do not work in non-reactive / pure code** — a `let m = [:]; m = m.insert(k,v); return m.size` emits **raw** `m.insert(...)` / `m.size` (method/property syntax) but the runtime map API is **free functions** (`_scrml_map_insert(m,k,v)`) and the HAMT struct carries `.count`, not `.size` → **runtime `TypeError: m.insert is not a function`**. Compiles clean (`node --check` passes — valid JS *syntax*), crashes at runtime (silent). **PA-VERIFIED S225** (emit inspection + the free-fn API + `.count`-not-`.size` = a deductive proof; the §59 "Implemented/R26-verified" banner only ever exercised REACTIVE maps). **Gates the compiler-in-scrml Road-B pure-fold dogfood** (the DG-builder color-map/reader-sets + the lexer accumulators are all non-reactive local maps/sets) — fix this before any pure-fold compiler work. MED, but Road-B-blocking.

**Root cause:** the map/set method lowering keys on the **`@`-cell sigil (reactivity)** + the `ctx.mapVarNames`/`ctx.setVarNames` registries (which collect only `@`-prefixed REACTIVE cells), NOT on the **type**. A non-reactive local map (`let m`, no `@`) isn't in those registries AND the `@`-prefix gate excludes it → the method call falls through to raw emission. (The reactive `@m.insert` path lowers correctly to `_scrml_map_insert(get("m"),…)`.)

**Shared ingestion:** the §59 map/set method-lowering dispatch in codegen — `emit-expr.ts` `MAP_METHOD_HELPERS` (emitCall), `emitMember` (`.size`), `emitIndex` (bracket-read `m[k]`), all `@`-gated; the `collectMapVarNames`/`collectSetVarNames` collectors; the name-set threading via `scheduling.ts`. coreFiles: `compiler/src/codegen/emit-expr.ts` (the 3 gates), `emit-functions.ts:446` + `emit-reactive-wiring.ts:290` + `emit-event-wiring.ts:532` (`collectMapVarNames`/`collectSetVarNames`), `scheduling.ts:378` (threads the sets). Pattern precedent: `emit-logic.ts:2391` — "codegen keys on the collected name-set."

**Brief reminders:** **VERIFY-FIRST** (compile the repro, confirm the raw `m.insert`/`m.size` emit + the runtime throw). The fix has two parts: **(1)** extend the collectors to gather **non-reactive local** map/set bindings — **scope-aware** (locals are fn/block-scoped, unlike module-level reactive cells): seed from `= [:]`/`= [k:v]` map literals, `: [K:V]` / `set[K]` annotations, and map/set-returning exprs/reassignments; **(2)** relax the `@`-prefix gate in `emitCall`/`emitMember`/`emitIndex` to fire for those local names, emitting the receiver as the **bare local** (`m`, not `_scrml_reactive_get("m")`). The crux/risk is the scope-awareness (per-function tracking) — SYM annotations are unreliable at codegen (codegen re-parses exprs), so use a local-scope syntactic collector mirroring the `mapVarNames` pattern. **Set rides the same fix** (set desugars to map; `.add`/`.elements()`/`.has`/algebra on non-reactive local sets break identically). R26: compile a pure `fn` using a local map AND a local set → `node --check` + **RUN it** (must not TypeError) + the full §59 map/set test suite + full `bun run test`. Adversarial: nested local maps, a local map passed to / returned from a fn, single-use vs reassigned, ordered maps. **The §59 banner + the §59.12 set note get an "incl. non-reactive locals" re-verification.** ⚠️ shares `emit-expr.ts` with ss50 (unary-paren / arrow-truncate) — sequence/reconcile if both in flight.
