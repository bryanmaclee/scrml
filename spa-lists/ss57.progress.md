## Escalations (impl#1-vs-SPEC divergences — for PA ruling, NOT decided by sPA)

- **E1 — min/max NaN on string fields.** On a STRING (`type=text`) form field, `min(n)`/`max(n)`
  ALWAYS fire `MinFailed`/`MaxFailed` (NaN comparison) regardless of value, while
  `gt/lt/gte/lte` coerce the string correctly. impl#1 min/max-vs-comparison INCONSISTENCY vs
  §55.1's "numeric" intent. Item-1 sidesteps it via the canonical numeric-field form
  (`type=number`, where min/max work); the string-field bug is flagged, not enshrined.
  → PA: is this an impl#1 coercion bug, or should min/max on a string cell be a compile-time
  type error (E-TYPE-031 family)? Either way §55.1 + impl need reconciliation.
- **E2 — LengthFailed payload shape.** §55.9 declares `LengthFailed(predicate: string)` but
  impl#1 snapshots `predicate:{op,value}` (structured object). Standing SPEC-vs-impl
  payload-ENCODING divergence — asserted-as-impl#1 in the cases (parallel to the §19.9.1
  server-error-wire precedent the conformance README cites). `PatternMismatch.re` also
  snapshots as `{}` (a regex has no JSON form — serialization artifact, not a semantic claim).
  → PA: amend §55.9 to the structured predicate shape, or is `predicate:string` the intended
  contract (impl bug)?
- **E3 — §55.10 message-resolution chain RUNTIME half largely UNIMPLEMENTED.** impl#1 accepts
  the syntax for Level-1 inline overrides (`req("msg")` / `length(>=2,"msg")`) and Level-2
  `registerMessages(...)` but IGNORES both at render — `messageFor` always falls through to the
  Level-3 shipped default. The Level-4 `<match on=@field.errors[0]>` escape hatch CRASHES at
  runtime (`_scrml_reactive_get("signup").name` undefined). So only Level 3 renders (impl-freedom
  text), and the "inline > registered > default" precedence CANNOT be conformance-pinned today.
  Item 5 shipped the CODES half (colon/dynamic reject + static-accepted); the RT half is
  impl-gated (parallel to ss56's harness-gated timers). Secondary: `E-VALIDATOR-INLINE-DYNAMIC`
  fires for a cell/identifier message arg but NOT for a template-literal `${}` message, contra
  §55.10 Edge F's "${} inside the message" wording.
  → PA: is the message-resolution chain (Levels 1/2/4) an unshipped surface (spec ahead of impl,
  a freeze-bar gap) or a set of impl bugs? This is a CLAIMED §55.10 pillar with no runtime.
