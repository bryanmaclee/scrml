## Findings (empirical, against trucking-dispatch, post-W2)

**F1 — W3-codegen built + working (headline; Rule 4).** See TL;DR. The chunk *files* and *manifest* are real, non-empty, deterministic.

**F2 — Chunks are mount-marker DESCRIPTORS, and NOTHING LOADS THEM (the W3↔W4 gap).** The augmented `dispatch/board.html` still loads the **full monolithic `.client.js` chain** (`board.client.js`, `../app.client.js`, every component `.client.js`) via `<script src>`. The initial chunk file is **never loaded as a script** — it is only named inside the inline `window._SCRML_CHUNKS` manifest object. `_scrml_chunk_mount(id, tag)` merely does `_SCRML_MOUNTS[id] = tag` (a registry write, no payload shrink). **Net: the "feel of performance" payoff is NOT yet realized — both the monolith AND the chunks emit; nothing is smaller.** This is exactly the W4 boundary SCOPE.md §2 flagged ("W4 — runtime tiered-chunk loader ... where 'feel of performance' becomes observable. May fold into W3.").

**F3 — Only `_anonymous` role is split.** Trucking has driver / dispatcher / customer roles (`pages/driver/**`, `pages/dispatch/**`, `<auth role=...>`), but the manifest emits ONLY the `_anonymous` variant for all 21 EPs. Role-keyed splitting — the entire *point* of "ship the role's surface" — does not yet happen. This is upstream RS (role projection), NOT W3-codegen's job.

**F4 — tier1/tier2 are empty everywhere (`serverFnNodeIds === 0`).** All prefetch tiers admit nothing → empty payload → no tier file written. Confirms SCOPE.md's note: the N≥1 interaction delta + server-fn projection is **Component-3, a later wave**. The initial chunk (N=0 playable surface) is the only populated tier.

**F5 — Manifest references tier files that aren't written (KNOWN-DEFERRED, not a surprise).** `chunks.json` lists `tier1`/`tier2` URLs (shared empty-input hash `00kas4tt`) for every EP, but those files are skipped on write (empty payload). `route-splitter.ts:1961-64` documents this as deliberate A-4.1 behavior ("skips writing empty chunk files but still surfaces a manifest entry") with the disposition "deferred separately." Becomes a real 404 risk **once a runtime loader (W4) follows those URLs** — must be resolved (omit empty tiers from manifest, OR runtime guards) when W4 lands.

## PARKED design forks (PA/user ruling — do NOT decide at sPA layer)

1. **W3↔W4 boundary: who shrinks the initial payload?** Today chunks are a parallel descriptor artifact; the monolith still ships (F2). The runtime loader (W4) is what makes the page load the initial chunk *instead of* the monolith. Forks: (a) the chunk REPLACES the per-route `.client.js` (initial chunk is self-contained executable, monolith dropped); (b) the chunk stays a mount-MANIFEST and the runtime selectively loads `.client.js` fragments it names; (c) fold W4 into W3 vs keep separate. **This is the load-bearing fork — it decides whether A-4 ever delivers the "feel of performance."**
2. **Role projection (F3).** Only `_anonymous` ships. When/how does RS emit per-role playable surfaces (driver/dispatcher/customer)? This is upstream of codegen (Component-3/4 role keying). Sequencing: before or after W4?
3. **Empty-tier manifest disposition (F5).** Omit empty tier1/tier2 from `chunks.json`, OR keep them + have the W4 loader guard. Couples to the runtime loader design.
4. **Component-3 N≥1 interaction projection (F4).** `serverFnNodeIds=0` → tiers empty. Big wave; out of W3-codegen scope. Confirm it's the next analytical wave after roles.
5. **PROPOSED SCOPE.md currency correction (PA to apply at re-integration).** Row 30 `A-4 codegen splitter ❌ NOT built` → `✅ BUILT (A-4.1..A-4.7, S91; verified non-empty post-W2, S221)`. Row 31 runtime loader stays `❌`. Reframe the W3 estimate (~60-120h) — the codegen payload is DONE; remaining is W4 loader + roles + Component-3. (Left for PA; sPA does not edit PA scoping docs.)
