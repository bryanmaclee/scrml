## ESCALATIONS (PA rulings needed — NOT decided by sPA)

### #1 — `fail`-variant validation unimplemented in impl#1 (§19.3.3 / §19.4.2 vs impl)
**Divergence.** SPEC §19.3.3: "The variant specified in the `fail` statement SHALL be a valid
variant of that error enum type. A variant that does not belong to the declared error type
SHALL be a compile error (E-TYPE-001)." §19.4.2 restates it. **impl#1 emits nothing** for any
of:
- `fail MyError.Nonexistent("x")` — variant not declared on MyError → expected E-TYPE-001, got [] (only incidental W-PROGRAM-001)
- `fail Other.Whatever` inside `function f()! MyError` — a foreign enum's variant → expected E-TYPE-001, got []
- `fail MyError.Plain("oops")` where `Plain` is nullary — payload on a unit variant → got []
- `fail "just a string"` in `function f()!` — non-enum fail value → got []
Verified LIVE (chain wired to `onclick=go()`, W-DEAD-FUNCTION gone) — still silent.
grep confirms impl#1's `E-TYPE-001` is wired ONLY for §14.3 lifecycle transition-state checks
(`type-system.ts`), never for `fail`-variant validation. So this is a genuine impl gap, not a
dead-code skip.
**sPA action:** did NOT author a "no-code" case (would enshrine the gap against the spec — the
§19.9.1 wire-divergence precedent). Authored only the conformant E-ERROR-001 path.
**PA decision needed:** conformance-bug (impl must add E-TYPE-001 fail-variant checks) OR
spec-relaxation. Until ruled, the fail-variant reject sub-cases stay UNCOVERED.
Probes: `scratchpad/probe-fail{,2,3}.ts`.

### #2 — E-BATCH-001 not source-reachable in impl#1 (parser limitation)
**Impl gap (not necessarily a spec issue).** §8.9.2 / §19.10.5 mandate E-BATCH-001 when a
`!` handler carries BOTH an implicit coalescing envelope (2+ `?{}`) AND an explicit
`transaction { }`. But impl#1's parser produces a bare-expr / drops `?{}.run()` inside a
`transaction { }` block in function-body position: a source-level construction yields
E-SCOPE-001 + "statement boundary not detected" parse-drop, never E-BATCH-001. This is
DOCUMENTED in impl#1's own `compiler/tests/unit/batch-planner.test.js` §11/§13 (both bypass
the parser with a hand-built AST + direct `runBatchPlanner`). The conformance harness compiles
from SOURCE, so E-BATCH-001 (and the `.nobatch()`-resolves-E-BATCH-001 path) cannot be authored
until the parser lands `transaction { <sql> }` in fn-body position.
**PA decision needed:** file the parser gap as an impl bug (then re-author E-BATCH-001 as a
conformance case) — or accept a permanent unit-test-only carve-out for E-BATCH-001.
Plus a naming note: the list's `@nosql-tx` opt-out does not exist; the real opt-out is
`.nobatch()` (§8.9.5). Probes: `scratchpad/probe-batch.ts`, `scratchpad/probe-wbatch.ts`.
Also observed (incidental, not escalated here): `server function` surfaces
W-DEPRECATED-SERVER-MODIFIER — the explicit `server` modifier is deprecated in favor of `?{}`
auto-escalation (§12).
