# Lint triage on a real adopter app — 2 false-positive classes + 1 watcher gap

Context: `scrml-site` @ `a401fe9`, freshly rewired onto `scrml` v0.7.1 and fully conformant
(`scrml build` exits 0). We triaged the full warning surface. Most of it is **noise generated by two
false-positive classes**, which matters because noise at this volume trains an adopter to ignore the
lint channel entirely.

Separate from the nested-list reconcile bug already filed today
(`2026-07-22-1120-...-nested-list-reconcile-stale.md`).

---

## 1. `W-DEAD-FUNCTION` — the analysis does not traverse arrow-callback bodies (FALSE POSITIVE)

**Claim:** "Function `buildJsCellLines` has no callers, is not exported, is not server-annotated, and
is not referenced from markup."

**Reality:** it is called on `pages/index.scrml:385`. `enginesOf` likewise, on line 398. Both are
called from inside `.then(...)` arrow callbacks:

```
function loadArtifacts() {
    fetch(base + "manifest.json").then((r) => r.json()).then((m) => {
        fetch(base + m.clientJs).then((r) => r.text()).then((t) => {
            fetch(base + m.jsMap).then((r) => r.json()).then((map) => {
                let segs = parseMappings(map.mappings)
                @jsCellLines = buildJsCellLines(t, segs, map.x_scrml_kinds)   // <- "no callers"
            })
        })
        fetch(base + m.engineGraph).then((r) => r.text()).then((t) => {
            @engines = enginesOf(t)                                          // <- "no callers"
        })
    })
}
```

**Isolated the trigger.** In the same file, these are NOT flagged:
- `parseMappings` — called on the line directly ABOVE `buildJsCellLines`, same arrow body. Escapes
  because a same-named `export fn` exists in `lib/provenance.scrml`.
- `cellsForLine` — called from inside another `fn` body. Traversal into `fn` bodies works.
- `flagshipList` — called at the top level of `function init()`. Traversal into plain `function`
  bodies works.
- `toLines` — called 3× and ONLY from inside arrow callbacks, i.e. exactly the broken case. It
  escapes solely because `components/{source-pane,output-tabs}.scrml` also `export fn toLines`.

**Decisive probe:** rename the local `toLines` → `toLinesUniqueProbe` (unique name, all 3 call sites
updated, nothing else changed) and it is immediately flagged dead:

```
Function `buildJsCellLines`
Function `enginesOf`
Function `toLinesUniqueProbe`      <- newly flagged, still called 3x
```

So: **a `fn` whose only call sites are inside arrow-function callback bodies is reported dead.** A
same-named export anywhere in the project masks it, which is why this reads as random.

Arrow callbacks in `.then()` chains are the normal shape for fetch-driven client code, so any app
that loads data this way will see phantom dead-function warnings on exactly its data-plumbing
helpers.

## 2. `W-TAILWIND-UNRECOGNIZED-CLASS` — does not see classes defined in the file's own style block (FALSE POSITIVE)

17 warnings in dev, every one on a class **we define ourselves** in a `#{}` CSS block in the same
project:

| class | defined in |
|---|---|
| `viewer-nav`, `viewer-nav-brand`, `viewer-nav-sep`, `mono`, `built-in-pill` | `app.scrml` style block |
| `showcase-grid`, `showcase-live`, `showcase-iframe`, `showcase-live-note`, `showcase-panes`, `pane-label` | `pages/index.scrml` / `components/showcase-layout.scrml` |

The message suggests the class "is misspelled, is a Tailwind arbitrary-value class, or is a custom
class defined elsewhere" and recommends "drop a `#{}` CSS shim block with the rules written by hand"
— which is **already what we did**. The lint appears not to cross-reference author-defined selectors
before firing.

Suggested fix: suppress when the class name matches a selector defined in any `#{}` block in the
compilation unit. Custom classes alongside Tailwind utilities is a completely ordinary shape.

## 3. `scrml dev` does not recompose page HTML when the SHELL (`app.scrml`) changes

**Not** the general watcher bug — we re-verified today that editing a **page** hot-recompiles
correctly (edit reached `dist/` + the served page within 12s). This is narrower and survives:

Editing the program shell `app.scrml` — specifically adding `<outlet/>` inside `<main>` — recompiles
without error, and the *lint* correctly stops emitting `W-OUTLET-ABSENT-SOFT-NAV-DISABLED`, but the
already-emitted **page** HTML is never recomposed. `dist/index.html` keeps the old bare `<main>`:

```
fresh `scrml build`   ->  <main> … <div data-scrml-outlet tabindex="-1">   ✅
running `scrml dev`   ->  <main>                                           ❌ (stale)
rm -rf dist + restart ->  <main> … data-scrml-outlet                       ✅
```

Consequence: a shell edit appears to do nothing. We chased this for a while assuming the outlet fix
had failed, when the emitted page just had not been recomposed. Page-level composition presumably
needs to be invalidated when the shell changes.

## Lower-severity note — `E-ROUTE-001` on plain array indexing

14 emissions (7 sites) of "Computed member access detected in expression `out.push({ n: i + 1, text:
raw[i] })` … Use a direct property access (e.g. `row.fieldName`) to ensure correct route placement."

All 7 are ordinary array/dict indexing in pure client-side helpers — `raw[i]`, `lines[gi]`,
`byLine[k]`, `fwd[srcKey]`, `rev[genKey]`. Our build wires **0 server routes**, so placement is
unambiguous and the warning is inert. The suggested remedy also cannot apply: you cannot iterate an
array without indexing it, and `row.fieldName` is a different shape entirely.

Not blocking us — flagging it because the route-placement heuristic seems to fire on any computed
access, including ones with no route-placement question attached.

## What we changed on our side

- Added `<outlet/>` inside `<main>` — soft nav now enabled (`W-OUTLET-ABSENT` cleared, runtime carries
  the nav code). Thanks for that warning; it was correct and actionable.
- 21 `W-STYLE-CONFLICT-POSSIBLE` accepted as advisory. They fire on our `.tok-*` syntax-highlight
  classes, which are mutually exclusive by construction (`lineClass()` returns exactly one) but the
  compiler can't prove that. No action requested.
