# A1b B12 — Rule 4 audit (pre-dispatch)

## §0 Scope

B12 = "Auto-synthesized validity surface — per-field — same per validator-tagged child cell. Builds on B11" (SCOPE §4.3 row B12, est 3-4h, locks L11).

This audit reads §55.6 (per-field surface), §55.7 (synthesized-property semantics), §55.11 (cross-field validation), §6.11 (forward stub), and flags drift BEFORE the B12 dispatch encodes the SCOPE wording into a brief. B12 builds on B11; this audit assumes B11 has shipped per the B11 audit's expanded brief.

## §1 Findings

### §1.1 [SUBSTANTIVE — SCOPE NARROWS THE SPEC] No-validator fields still get the surface

**SCOPE B12 row:** "same per validator-tagged child cell"

**SPEC §55.6 (line 24444-24447, "No-validator field"):**
> "Per L11 Edge B, a per-field surface exists EVEN when the field has no validators — `@signup.someUnvalidated.isValid` is trivially `true`; `errors` is `[]`. This is the predictability rule (§55.5): field-level access works regardless of whether validators are declared."

**Drift analysis:** the SCOPE row's "per validator-tagged child cell" wording implies surface synthesis is conditional on the child having validators. The spec is the opposite: per-field surface is unconditional for ALL fields of a compound parent (subject to B11 having synthesized the compound-level surface, which it does for ALL compound parents per the B11 audit §1.1).

This is the SAME shape of drift as B11 audit §1.1 — SCOPE narrows what the spec mandates. B12's expansion: synthesize per-field surface for EVERY field of every compound parent, not just validator-tagged fields. The trivially-valid case (no validators on this specific field) gets `isValid: true`, `errors: []`, `touched: false`.

**Required B12 implementation expansion:** walk every CHILD of every compound parent (all fields, regardless of validator presence). Register per-field synth cells (`isValid`, `errors`, `touched`) under the field's name within the compound's `_scope`.

**Cost impact:** trivial — same logic as the validator-tagged path, just default values. Maybe 15-30min added.

### §1.2 [BOUNDARY CLARIFICATION] `submitted` is compound-level only — B12 must NOT extend it

**SPEC §55.7 (line 24458):**
> "**`submitted`** Compound-level. Becomes `true` on first submit-form attempt. Once true, never reverts (until `reset`)."

**B12 must NOT synthesize `@signup.name.submitted`** — that property exists only at compound scope. B11 already synthesizes `@signup.submitted`; B12 leaves that alone.

This is not a SCOPE drift — SCOPE says "same per validator-tagged child cell" without enumerating properties. But B12's brief MUST explicitly enumerate: B12 synthesizes `{isValid, errors, touched}` per field, NOT `submitted`. (B11 audit §2 item #6 already calls this out for B11.)

### §1.3 [SPOT-CHECK PASS] Property type shapes match §55 (not §6.11 stub)

**SPEC §55.6 (line 24432-24435):**
> ```
> @signup.name.isValid     : boolean
> @signup.name.errors      : [...errorTags]      (array of ValidationError tags for this field)
> @signup.name.touched     : boolean
> ```

**§6.11 stub (line 4973):** `@x.field.error : string | not` — singular and string-typed.

**Drift analysis:** §6.11 stub is acknowledged-stale. §55.6 is canonical. B12 must use the §55 forms — `errors` (plural, array of enum tags), NOT `error` (singular, string).

This is a Rule-4 follow-through from the B11 audit §1.2; recorded here for B12 brief completeness.

**SPEC follow-up flagged (same as B11 audit §4):** §6.11 stub footnote referencing §55 canonical. Single spec-prose commit covers both audits.

### §1.4 [WAVE-ORDERING] B12 extends B8's PASS 6 walker for per-field E-SYNTHESIZED-WRITE

**B11 audit §1.3 + B8 SHIPPED state:** B11 fires `E-SYNTHESIZED-WRITE` at compound scope by extending B8's PASS 6 walker.

**B12 implementation guidance:** B12 extends the SAME walker dispatch path with per-field receiver chains. A write to `@signup.name.isValid = true` resolves through the receiver chain:
- `@signup` → resolves to compound parent (B5 `_cellKind: "compound-parent"`).
- `.name` → static property; resolves to compound child cell.
- `.isValid` → the synth-property name in the dispatch set.

The walker dispatch added by B11 already handles compound-scope writes (`@signup.isValid = true`). B12 extends the receiver-chain resolution to support per-field paths (`@signup.name.isValid = true`).

**Receiver-chain resolution reuse:** B8 audit §1.2 case 3 (in-compound derived sub-cell) already handles "receiver chain extends through compound parent." B12 reuses this exact pattern. Per primer §13.7 B8 specifics: `findDeepestRegisteredOnPrefix` walks prefixes longest→shortest. B12's per-field synth-cells are registered at depth-2 (compound + field-name); the prefix walk will find them.

### §1.5 [REUSABILITY — B7 DEP-GRAPH CONSUMER] Cross-field validation deps

**SPEC §55.6 (line 24437-24443):**
> "Reactive recomputation:
>   - A change to `@signup.name` recomputes that field's surface.
>   - A change to a cell referenced in a cross-field predicate arg (e.g., `@signup.password` referenced from `<confirm req eq(@signup.password)>`) recomputes the dependent field's surface.
>   - The compound's `@signup.isValid` recomputes whenever ANY field's surface changes."

**SPEC §55.11 (line 24608-24635, cross-field validation):** predicate args may reference other state cells via canonical `@name` form. Deps are tracked via the standard reactive dependency tracker (§31.4).

**B12 implementation guidance:**
- Per-field `isValid` reactive dep: the field's own cell value + the field's validator chain output.
- Per-field `errors` reactive dep: same.
- Per-field cross-field deps: when a field's validator has a predicate arg referencing another cell (`eq(@signup.password)`), B12 emits a dep-edge from THIS field's surface to the referenced cell.
- B12 emits dep-edges into B7's existing dep-graph; no new walker logic.
- B11 already emits the compound-rollup dep-edge (compound `isValid` ← every field's `isValid`); B12 emits the per-field dep-edge (field `isValid` ← validator output ± cross-field deps).

This is the natural consumer of the B7-reusability promise (audit §1.4).

### §1.6 [SEMANTICS GUIDE — §55.7] Per-field `touched` runtime hook

**§55.7 line 24457:**
> "**`touched`** Becomes `true` on first interaction with the field — defined as ANY of: `bind:value` change, `bind:checked` change, OR first focus-out. The most-permissive trigger is chosen so the surface 'feels right' with idiomatic UI. Per-field timing. Once true, never reverts (until `reset` — §55.13)."

**B11 audit §1.5** already noted `touched` and `submitted` need runtime-hook annotations. B11 records hook requirements on synth-cell records. B12 records the SAME requirements on per-field synth-cell records, but the hook target is the per-field cell, not the compound.

**B12 implementation guidance:** for each per-field `touched` synth-cell, record annotation:
- Hook source: the field's bind:value or bind:checked target (if any) + the field's focus-out event.
- Latch behavior: elevate to `true` exactly once.
- Reset target: `reset(@signup.name)` reverts; `reset(@signup)` reverts the whole compound (each child resets transitively).

A1c codegen consumes these annotations. B12 (A1b) does not emit the actual runtime hooks.

### §1.7 [DEPENDENCY] B12 sequential after B11

**SCOPE §4.6 line 268:** "B12 | B11"

**SCOPE §4.7 (Wave 3 line 320):** "Wave 3 (validator + surface): B9 → B10 → B11 → B12 → B13 (sequential)"

**B12 strict sequential after B11.** No parallelism with B11 — B11 births the synth-cell registry; B12 extends it.

**Could B12 parallelize with B13 (E-DERIVED-WITH-VALIDATORS) once B11 lands?** B13 walks derived state-decls with validators (different code region from B12); could run in parallel with B12 if both branch from B11's tip. Worth flagging at B11-landing time — not B12-audit territory.

## §2 B12 dispatch brief — required additions beyond SCOPE row

When PA writes the B12 dispatch brief, the following MUST be in the brief:

1. **Per-field surface unconditional** for ALL fields of compound parents (NOT just validator-tagged ones). No-validator field gets trivially-valid defaults per §55.6 / L11 Edge B.

2. **Three properties per field, NOT four.** B12 synthesizes `{isValid, errors, touched}` per field. `submitted` is compound-level only (per §55.7) — do NOT extend to per-field.

3. **Type shapes per §55** (NOT §6.11 stub): per-field `errors` is plural array of `ValidationError` enum tags.

4. **E-SYNTHESIZED-WRITE walker extension:** B12 extends B11's dispatch path in B8's PASS 6 walker with per-field receiver-chain resolution. `findDeepestRegisteredOnPrefix` (already in B8) handles the depth-2 lookup automatically once per-field synth-cells are registered.

5. **Cross-field deps consume B7's dep-graph.** B12 emits dep-edges for cross-field predicate arg references; B7's existing infrastructure propagates. No new walker logic.

6. **`touched` runtime-hook annotations** on per-field synth-cells (same shape as B11's compound-level annotations). A1c codegen emits the hooks. Reset target is the field cell or the compound (transitively).

7. **Sequential after B11.** B12 dispatch fires only after B11 lands. Survey gate verifies B11's synth-cell registry API supports per-field extension.

8. **Phase-0 survey gate:** confirm (a) B11's synth-cell registry exposes a "register child cell's synth-properties" API, (b) B8's PASS 6 walker extension by B11 covers compound scope cleanly + supports the B12 per-field extension, (c) B7's dep-graph public API supports B12 emitting cross-field edges.
