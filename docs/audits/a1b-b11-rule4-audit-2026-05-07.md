## §1 Findings

### §1.1 [SUBSTANTIVE — SCOPE NARROWS THE SPEC] No-validator compounds still get the surface

**SCOPE B11 row:** "for compound-with-validators, synthesize `.isValid`, `.errors`, `.touched`, `.submitted` cells"

**SPEC §55.5 (line 24415-24418):**
> "**No-validator compounds.** When a compound has NO validators, `isValid` is trivially `true`; `errors` is empty per-field; `touched` and `submitted` exist as conceptual empty structures. Predictability over namespace savings — applications can check `@form.isValid` without first asking 'does this compound have any validators?'"

**Drift analysis:** the SCOPE row implies the surface is conditional on "at least one field has a validator." The spec is the opposite: compounds ALWAYS get the surface; the trivially-valid case is part of the predictability invariant (L11 Edge B's compound-level analog).

**However** — there is a real conditional in the spec, just at a different level:

**SPEC §55.5 (line 24420-24425, "Single-value Tier-1 cells DO NOT get the auto-namespace"):**
> "Per L11 Edge A, a top-level `<count req min(0)>` cell does NOT synthesize `count.isValid` / `count.errors` — those properties are available only on COMPOUND cells (cells with internal field structure)."

So the actual decision tree B11 must implement:

| Cell shape | Surface synthesized? |
|---|---|
| Single-value Tier-1 (e.g., `<count req>`) | NO — L11 Edge A |
| Compound, NO field validators (e.g., `<form><name/><email/></>`) | YES (trivially-valid surface) — §55.5 predictability rule |
| Compound, ≥1 field validators (e.g., `<form><name req/><email/></>`) | YES (full surface) |

**Required B11 implementation expansion:** synthesize the surface for ALL compound parents (B5's `_cellKind: "compound-parent"`), not just compounds-with-validators. The trivially-valid case (no validators on any child) still needs `isValid: true`, `errors: {}` (empty object map), `touched: {}` (empty), `submitted: false`.

**Cost impact:** SCOPE estimate 4-6h. Adding the no-validator-compound case is mostly a default-value branch; ~30-60min added. Total maybe 5-7h.

**B11 implementation guidance:**

- B11 builds the dep-edge structure CONSUMING B7's generic infrastructure (ADD edges to the existing dep-graph; rely on existing fixed-point propagation).
- No new walker logic for dep tracking — B11 emits dep-edges, B7 already wires them.

### §1.5 [SEMANTICS GUIDE — §55.7] Update timing per property

**§55.7 line 24449-24461** specifies update timing for the four synthesized properties:

| Property | Update timing | Default value (no validators) |
|---|---|---|
| `isValid` | Reactive — recomputes whenever any contributing validator's inputs change. | `true` |
| `errors` | Reactive — recomputes per-field on cell change or cross-field dep change. | `[]` (empty array) per-field; `{}` (empty object) compound-level |
| `touched` | Becomes `true` on first interaction with the field — defined as ANY of: `bind:value` change, `bind:checked` change, OR first focus-out. Per-field timing. Once true, never reverts (until `reset` — §55.13). | `false` |
| `submitted` | **Compound-level.** Becomes `true` on first submit-form attempt. Once true, never reverts (until `reset`). | `false` |

**B11 implementation guidance:**
- `isValid` and `errors` are pure reactive derivations — wire them as derived cells with the appropriate dep-edges; existing reactive infrastructure handles the rest.
- `touched` and `submitted` have RUNTIME-event-driven update timing — they are NOT pure reactive derivations. They need:
  - A runtime hook for the touch trigger (bind: change or focus-out).
  - A runtime hook for the submit trigger (form's submit handler).
  - Both hooks elevate the cell to `true` exactly once (latch behavior).
  - Reset via `reset(@cell)` keyword (§55.13, L18) reverts.
- B11 (A1b — symbol-table and registration) does NOT emit the runtime hooks — that's A1c codegen. B11 records the runtime-hook requirement on each synthesized cell so A1c knows what to emit.

**`submitted` is COMPOUND-LEVEL ONLY** — do NOT synthesize per-field `@signup.name.submitted`. (Important boundary; per §55.7 table.)

## §2 B11 dispatch brief — required additions beyond SCOPE row

When PA writes the B11 dispatch brief, the following MUST be in the brief:

1. **Surface synthesis is unconditional for compound parents.** Single-value Tier-1 cells (L11 Edge A) → no surface. ALL compound parents (B5 `_cellKind === "compound-parent"`) → surface synthesized. No-validator compounds get trivially-valid defaults per §55.5 predictability rule.

2. **Type shapes per §55, NOT §6.11 stub.** Compound `errors` is `{ fieldName: [...errorTags] }` (object map); enum tags are `ValidationError` per §55.9; per-field `errors` is plural array of enum tags (NOT singular `error: string`).

3. **B11 fires `E-SYNTHESIZED-WRITE`** (compound scope) by extending B8's PASS 6 walker with a fourth dispatch path keyed on synth property names `{isValid, errors, touched, submitted}`. Walker structure is prepared per primer §13.7 B8 specifics. Per-field scope is B12's extension.

4. **Cross-field predicate-arg deps consume B7's generic dep-graph.** No new walker logic; emit dep-edges via the existing infrastructure (per primer §13.7 B7 specifics). The compound's `isValid` rolls up via dep-edges from each field's `isValid`.

5. **`touched` and `submitted` need runtime-hook annotations.** B11 records hook requirements on the synth-cell records; A1c codegen emits the actual hooks. Touch trigger: ANY of bind:value change / bind:checked change / first focus-out (§55.7). Submit trigger: form's submit handler. Both have latch semantics with `reset(@cell)` revert.

6. **`submitted` is compound-level ONLY** — do NOT synthesize per-field. (B12 must not extend this property to per-field.)

7. **B5's `_cellKind === "compound-parent"` is the trigger predicate.** B11 walks state-decls with this annotation; for each, registers the four synth-cells in the compound's `_scope`.

8. **Phase-0 survey gate:** confirm (a) B5's `_cellKind` annotation is reliable (has it been verified across the test corpus?), (b) the symbol table API supports adding entries to a compound's scope post-B5, (c) B7's dep-graph public API supports adding edges from B11's pass.
